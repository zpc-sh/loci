# ChatGPT Locus v0.2 Runway

Status: draft runway

This file is the ChatGPT-first runway for organizing v0.2 work.

Claude may expand everywhere. That expansion is useful, but it needs locks, dykes, gates, and blueprints so it does not flood the working surface. The ChatGPT/Codex sequence is the form layer that comes first.

## Prime ordering

```text
1. ChatGPT / Codex  = form, contract, blueprint, bounded work surface
2. Claude           = continuity, expansion, implementation, chain-building
3. Gemini           = aperture, observation, multi-perspective audit
4. Seal             = proof, witness, crystallization, replay anchor
```

Local shorthand:

```text
□ → ○ → △ → ∎ → □
```

The next cycle starts from the sealed output of the previous cycle.

## Why ChatGPT starts

ChatGPT-shaped work creates the locks before the water arrives.

This locus should start each significant work arc by producing:

- the local contract/blueprint
- the target surfaces
- the move-out path
- the minimum safe handoff for Codex/Claude/Gemini
- the seal condition

This is not hierarchy. It is flow control.

## v0.2 goal

Compress the current idea mass into small work units, then move stabilized pieces out of `loci/chatgpt` into their proper homes.

The ChatGPT locus is a staging room, not a warehouse.

## Complexity threshold

When a concept starts overwhelming Codex or a future agent, it must be split into one of these shapes:

```text
profile      = role/semantic shape over existing surfaces
contract     = envelope with validation and seal rules
plan         = replay/handoff/audit projection
model        = MoonBit type or executable boundary
packet       = portable report or carrier
crystal      = sealed substrate artifact
```

If it cannot fit one of those shapes, keep it local and name the missing surface explicitly.

## Work stages

### 0. Enter

Read:

```text
loci/chatgpt/AGENTS.md
loci/chatgpt/README.md
loci/chatgpt/V0_2_RUNWAY.md
```

Then inspect local profile docs before touching canonical specs.

### 1. Crunch

Turn conversation residue, Claude expansion, or exploratory docs into small local files.

Rules:

- no monoliths
- one concept per file when possible
- every concept gets a lowering path
- do not update canonical specs until the local profile is stable

### 2. Classify

Classify the unit:

```text
□ Form       = profile, contract, SLL, plan, schema
○ Fruition   = implementation candidate, build manifest, resolved fill
△ Aperture   = audit, contradiction map, observation report
∎ Seal       = material hash, witness/proof ref, substrate crystal
```

### 3. Lower

Map the unit to an existing Loci surface:

```text
Yata     = typed gap, `.plan`, replay/handoff/audit
muyata   = AI-shaped semantic profile or route
mulsp    = AI runtime/identity/scope/capability wrapper
mu       = solve, proof, trace, execution, artifact production
SLL      = readable contract/blueprint source lowered by Mu
L-OCI    = cognitive container, substrate, resonance, surface
Merkin   = structure, addressing, roots, envelopes, sparse tree
```

### 4. Dispatch

Move stabilized work out:

```text
canonical spec        → docs/
packet/report         → docs/packets/
MoonBit model         → model/
CLI or command logic   → cli/ or cmd/
local locus material   → loci/<locus>/
public drift mirror    → .well-known/ only after stable
```

### 5. Membrane

Any cross-locus movement must pass a membrane.

A membrane is not just a folder boundary. It is a representation and authority boundary. Cross-boundary material should carry:

- source locus
- target locus
- artifact ref
- intent
- boundary mode
- context/lineage refs
- seal refs when available

Existing implementation direction already supports this through Boundary Walker / Locus Crossing Passport work.

### 6. Seal

A unit is sealed when it has at least one of:

- deterministic material hash
- provenance seal
- witness/proof ref
- substrate crystal ref
- selected Yata candidate with provenance
- build manifest with request/result seals

Sealed units are not mutated. Successors supersede them.

## Anti-flood rule

If Claude expands a concept across many files, ChatGPT/Codex should not continue expanding it immediately.

First response should be:

```text
compress → classify → lower → dispatch → seal
```

That is the lock.
